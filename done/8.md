# 8 - Connect GitHub Actions to k3s cluster via Tailscale

# Description

The GitHub Actions deploy workflow (`deploy.yml`) cannot reach the k3s cluster
because the runner has no access to the libvirt NAT network the cluster runs
on. The fix is to use Tailscale to join the runner to the same tailnet as the
k3s node during each workflow run.

## Approach

1. Install Tailscale on the k3s host node.
2. Add the `tailscale/github-action` step to the workflow — it joins the
   ephemeral runner to the tailnet using a reusable auth key.
3. Update the `KUBECONFIG` GitHub secret to point to the k3s node's Tailscale
   IP instead of `127.0.0.1`.

## Affected files

- `.github/workflows/deploy.yml` — add Tailscale connect step after Checkout
- `docs/self-hosted-runner.md` — document Tailscale setup on the k3s node and
  the new `TAILSCALE_AUTHKEY` secret; document Tailscale as alternative to SSH
  tunnel for local dev
- `KUBECONFIG` GitHub secret — must be updated to use the Tailscale IP
  (manual operator step, not automated)

## Notes

- The Tailscale auth key must be **reusable + ephemeral**: reusable so repeated
  CI runs work without regenerating the key; ephemeral so runner nodes
  auto-expire from the tailnet after the job completes.
- The k3s node only needs to be a Tailscale peer (no subnet routing required)
  since the runner only needs to reach port 6443 on that node.
- The `KUBECONFIG` secret must have `server: https://<tailscale-ip>:6443`.
- Do NOT delete `scripts/tunnel.sh` — it remains useful for local port
  forwarding (ports 80/8080) outside of CI.

# PRD

## Introduction / Overview

The GitHub Actions deploy workflow (`deploy.yml`) runs on a self-hosted runner
that cannot reach the k3s API server because the cluster node lives on a
libvirt NAT network that is not reachable from the runner's network context.
Tailscale will bridge that gap: the k3s node joins a tailnet, and each CI job
temporarily joins the same tailnet during the run, giving the runner direct
access to port 6443 on the k3s node.

## Goals

1. Allow the `deploy.yml` workflow to authenticate with the k3s API server
   without manual network configuration.
2. Require no persistent changes to runner infrastructure between jobs — each
   connection is ephemeral.
3. Document Tailscale as an optional replacement for the SSH tunnel used during
   local development.

## User Stories

- **As a CI/CD pipeline**, I need to reach `https://<tailscale-ip>:6443` so
  that `helm upgrade` and `kubectl` commands succeed during deployment.
- **As a developer**, I need a step-by-step guide to set up a Tailscale
  account, enroll the k3s node, and configure the required GitHub secret so
  the pipeline works end-to-end.
- **As a developer doing local work**, I want to know whether Tailscale can
  replace the SSH tunnel so I can skip the tunnel setup if I have Tailscale
  installed.

## Functional Requirements

1. **Tailscale account**: Create a free Tailscale account at `tailscale.com`
   and note the tailnet name.
2. **Install Tailscale on k3s node**: Install the Tailscale agent on the
   VM/host running k3s, enroll it in the tailnet, and note the stable Tailscale
   IP (e.g. `100.x.y.z`).
3. **Generate auth key**: In the Tailscale admin console, create an auth key
   that is **reusable** and **ephemeral**. Store it as the GitHub Actions secret
   `TAILSCALE_AUTHKEY`. A one-time-use key expires after the first job run; a
   reusable ephemeral key is required for repeated CI runs while still
   auto-expiring runner nodes from the tailnet.
4. **Add Tailscale step to workflow**: In `.github/workflows/deploy.yml`, add
   the `tailscale/tailscale-github-action` step immediately after the
   `Checkout` step, passing `authkey: ${{ secrets.TAILSCALE_AUTHKEY }}`.
5. **Update `KUBECONFIG` secret**: Document that the `KUBECONFIG` GitHub secret
   must be updated so `server:` points to `https://<tailscale-ip>:6443`
   instead of `127.0.0.1:6443`. This is a manual operator step.
6. **Update `docs/self-hosted-runner.md`**: Add a "Tailscale Setup" section
   covering steps 1–5, the new `TAILSCALE_AUTHKEY` secret, and a note
   explaining that if the developer also has Tailscale running locally with the
   k3s node enrolled, the SSH tunnel (`scripts/tunnel.sh`) is no longer needed
   for accessing the cluster API — document how to use the Tailscale IP
   directly with `kubectl` in that scenario.
7. **Do not delete `scripts/tunnel.sh`**: The script remains useful for local
   port-forwarding of the app (ports 80/8080) independent of the cluster API.

## Non-Goals

- Automating Tailscale installation on the k3s node (manual operator step).
- Configuring subnet routing or Tailscale exit nodes — only point-to-point peer
  access to port 6443 is needed.
- Rotating or managing the `TAILSCALE_AUTHKEY` secret automatically.
- Replacing the SSH tunnel for local app access (ports 80/8080) — only the
  cluster API access scenario is in scope.

## Technical Considerations

- Use `tailscale/tailscale-github-action@v2` (or latest stable).
- Ephemeral nodes auto-remove from the tailnet after the runner disconnects,
  keeping the tailnet clean.
- The k3s node's Tailscale IP is stable across reboots as long as it stays
  enrolled in the tailnet.
- The k3s TLS certificate is issued for the LAN IP, not the Tailscale IP. The
  `KUBECONFIG` secret will likely need `insecure-skip-tls-verify: true` as a
  pragmatic workaround, or the k3s node can be configured with
  `--tls-san <tailscale-ip>` at install time to include the Tailscale IP in
  the certificate SANs. Document both options.

## Success Metrics

- A push to `main` triggers the full `deploy.yml` pipeline and the
  `Deploy with Helm` step completes without a `connection refused` error.
- The runner node does not persist in the Tailscale admin console after the
  job completes.

## Open Questions

1. **Auth key expiry**: Should the reusable ephemeral key have a short expiry
   (e.g. 90 days) with a documented rotation procedure, or no expiry?
   → **Decided**: 90-day expiry with documented rotation procedure.
2. **TLS SAN**: Is `insecure-skip-tls-verify: true` in the KUBECONFIG
   acceptable, or should the k3s node be reconfigured with
   `--tls-san <tailscale-ip>` to avoid certificate warnings?
   → **Decided**: Use `insecure-skip-tls-verify: true`.

        ```bash
        sudo tailscale up
        ```
