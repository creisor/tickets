# 7 - Epic: MoonPay coding challenge

# Description

This is a coding challenge, the goal of which is to productionalize the deployment of a Next.js application.

See `./README.md` and `./DEVELOPMENT.md` for an overview.

There are two major requirements we must break into multiple tasks.

Requirement 1: Containerize the Application
  - Write a `Dockerfile` to containerize the application.
  - Ensure it follows best practices for a Next.js application.
  - Build and run the container locally to verify it works.

Requirement 2: Deploy the Application
- Deploy the application to **Kubernetes** (local k3s cluster)
  - Helm
  - Github Actions
  - Postgres DB will also have to run in Kubernetes
    - This also needs a Helm chart
    - We should run Postgres as a Statefulset with PVC for storage
- Ensure the solution is **as close to production-ready as possible**. Consider aspects like:
  - Security (run as non-root, minimal base image, no priveleged containers, shift-left package scanning, etc)
  - Scalability (autoscaling, etc.)
  - Reliability (deployments, probes, etc.)
    - note for probes: it looks like we will have to create health checks (a simple one and a database-connection-checking one)
- Demonstrate that the application is reachable and returns the _Latest Crypto Prices_.
  - e.g.: post deploy smoke tests

Other requirements:
- Use Terraform for any other infrastructure required by the application, if relevant.
- All Terraform, Helm, and any other new artifacts should live in this repo (as opposed to their own repo).

Notes about the Kubernetes cluster setup:
- The cluster is k3s running on another laptop in my lan
- The cluster runs on a private network (libvirt NAT), so we expose services via an SSH tunnel, similar to a bastion host pattern in production environments
- For simplicity, we can use `/etc/hosts` as local "DNS"
- The cluster runs open source Traefik
- K3s comes with Rancher's Local Path Provisioner (`storageClassName: local-path`) 

# PRD

## Introduction / Overview

This epic covers the end-to-end productionalization of the MoonPay DevOps Challenge application — a Next.js 16 / TypeScript app backed by PostgreSQL 17, displaying live cryptocurrency prices. The work involves containerizing the application, deploying it to a local k3s Kubernetes cluster via Helm and GitHub Actions CI/CD, and ensuring the result is as close to production-grade as possible. This will be presented to MoonPay interviewers on Tuesday, 2026-04-21.

## Goals

1. Produce a production-quality `Dockerfile` using multi-stage builds that minimizes the final image size.
2. Publish images to GitHub Container Registry (GHCR) via GitHub Actions on every push to `main`.
3. Deploy the Next.js app and a PostgreSQL 17 StatefulSet to a local k3s cluster using separate Helm charts.
4. Meet production-readiness standards: security, scalability (HPA), and reliability (probes, rolling deployments).
5. Verify the deployment end-to-end with a post-deploy smoke test.
6. Document and automate the SSH tunnel setup used both for local access and for GitHub Actions deployments.

## User Stories

- As a **developer**, I can build and run the containerized app locally with a single `docker build` + `docker run` so I can iterate quickly.
- As a **CI/CD pipeline**, I can build, scan, push, and deploy the app automatically on every merge to `main` so no manual steps are needed for deployment.
- As a **cluster operator**, I can install or upgrade the app and its database with `helm upgrade --install` so deployments are repeatable and version-controlled.
- As an **interviewer**, I can hit the app through the SSH tunnel and see the _Latest Crypto Prices_ page so I can verify the solution works end-to-end.
- As an **on-call engineer**, I can query `/api/health/live` and `/api/health/ready` to quickly determine whether the app and its database connection are healthy.

## Functional Requirements

### Requirement 1 — Containerize the Application

1. Write a multi-stage `Dockerfile` using `node:22-alpine` as both the build and runtime base image.
   - **Stage 1 (deps):** Install only production `node_modules` using `pnpm`.
   - **Stage 2 (builder):** Copy source, install all deps (including dev), run `pnpm build`.
   - **Stage 3 (runner):** Copy built output and production `node_modules` only; run as a non-root user (`nextjs`/UID 1001).
2. Set `output: 'standalone'` in `next.config.ts` to minimize the runtime image size (recommended approach for Docker/Kubernetes deployments).
3. The container must accept `POSTGRES_PRISMA_URL` as a runtime environment variable (not baked into the image).
4. The container must expose port `3000`.
5. The image must pass a Trivy vulnerability scan with no CRITICAL or HIGH CVEs in the final stage (shift-left scanning in CI).
6. The existing `docker-compose.yaml` `nextjs` service must continue to work with the new `Dockerfile`.

### Requirement 2 — Health Check API Routes

7. Add a `GET /api/health/live` route to the Next.js app that returns `200 OK` with `{"status":"ok"}`. Used as a Kubernetes **liveness** probe.
8. Add a `GET /api/health/ready` route that attempts a lightweight Prisma DB query (e.g., `SELECT 1`) and returns `200 OK` if successful, or `503 Service Unavailable` if the DB is unreachable. Used as a Kubernetes **readiness** probe.

### Requirement 3 — Postgres Helm Chart

9. Use the Bitnami `postgresql` Helm chart to deploy PostgreSQL 17.
10. Configure a `StatefulSet` with a `PersistentVolumeClaim` using `storageClassName: local-path` (k3s Local Path Provisioner).
11. Database credentials (username, password, database name) must be stored in a Kubernetes `Secret` and referenced by the chart values — not hardcoded in values files.
12. The chart values file must live in this repo at `helm/postgres/values.yaml`.

### Requirement 4 — App Helm Chart

13. Write a Helm chart for the Next.js app at `helm/app/`.
14. The chart must include:
    - `Deployment` with rolling update strategy.
    - `Service` (ClusterIP).
    - `Ingress` configured for Traefik (the k3s default ingress controller), using hostname `devops-challenge.local`.
    - `HorizontalPodAutoscaler` with `minReplicas: 1`, `maxReplicas: 3`, targeting CPU utilization.
    - `Secret` for `POSTGRES_PRISMA_URL` injected as an env var into the pod.
15. Containers must run as non-root (`runAsNonRoot: true`, `runAsUser: 1001`).
16. No privileged containers; drop `ALL` Linux capabilities.
17. Liveness probe: `GET /api/health/live`, initial delay 15s, period 10s.
18. Readiness probe: `GET /api/health/ready`, initial delay 10s, period 5s.
19. Resource requests and limits must be defined (reasonable defaults for a 3-node k3s cluster).
20. Prisma migrations must run as a Kubernetes `Job` annotated as a Helm `pre-upgrade` (and `pre-install`) hook, executing `pnpm db:migrate` before the app `Deployment` is updated. The Job must use the same `POSTGRES_PRISMA_URL` secret as the app. Set `hook-delete-policy: before-hook-creation` so old Job pods are cleaned up on each deploy.

### Requirement 5 — GitHub Actions CI/CD

21. On push to `main`, a GitHub Actions workflow at `.github/workflows/deploy.yml` must:
    1. Build the Docker image.
    2. Scan the image with Trivy; fail the pipeline on CRITICAL/HIGH CVEs.
    3. Push the image to GHCR using `secrets.GHCR_TOKEN`. Tag with the Git SHA and `latest`.
    4. Establish an SSH tunnel from the Actions runner to the k3s API server (see Requirement 6).
    5. Deploy via `helm upgrade --install` over the tunnel using a kubeconfig stored in `secrets.KUBECONFIG` (the Prisma migration Job runs automatically as a pre-upgrade hook).
    7. Run the smoke test script as the final step.
22. Image naming convention: `ghcr.io/<github-org>/devops-challenge:<sha>` and `ghcr.io/<github-org>/devops-challenge:latest`.

### Requirement 6 — SSH Tunnel / Cluster Access

23. Because the k3s cluster runs on a private LAN (libvirt NAT) and GitHub Actions runners have no direct network access, both local access and CI/CD deploys must go through an SSH tunnel to the host laptop.
24. Document the SSH tunnel setup in `docs/ssh-tunnel.md`, covering:
    - How the tunnel works (SSH port-forward from client → laptop → k3s node).
    - Command to open the tunnel for local use (forwarding the Traefik ingress port and the k3s API port).
    - The `/etc/hosts` entry needed to resolve `devops-challenge.local` locally.
    - How to export the kubeconfig and patch the server URL to `localhost` for use over the tunnel.
    - How to verify connectivity before running a deploy.
25. Provide a shell script `scripts/tunnel.sh` that opens the SSH tunnel in the foreground (ingress + API server ports).
26. For GitHub Actions, store the following as repository secrets:
    - `KUBECONFIG`: the kubeconfig with the server URL set to `localhost:<forwarded-port>`.
    - `SSH_PRIVATE_KEY`: the key used to SSH to the host laptop.
    - `SSH_HOST` / `SSH_USER`: the laptop's address and username.
    - `GHCR_TOKEN`: a GitHub PAT with `write:packages` scope.
27. The GitHub Actions workflow must establish the SSH tunnel as a background step before running `kubectl` or `helm`, and tear it down after.

### Requirement 7 — Post-Deploy Smoke Test

28. A smoke test script `scripts/smoke-test.sh` must:
    - `curl` `http://devops-challenge.local` and assert HTTP 200.
    - Check that the response body contains an expected string (e.g., `"Latest Crypto Prices"` or a known currency code like `"BTC"`).
    - Exit non-zero on failure with a descriptive error message.
29. The smoke test must be the final step in the GitHub Actions deploy workflow (run after `helm upgrade --install` completes).

### Requirement 8 — Terraform

30. Terraform is not required for this setup. Add a note to the repo README explicitly stating that Terraform was evaluated and determined to be out of scope for a local k3s environment.

## Non-Goals

- No GKE or cloud Kubernetes — local k3s only.
- No Terraform (unless a concrete need emerges).
- No TLS termination beyond Traefik defaults (no cert-manager).
- No external DNS — `/etc/hosts` is sufficient.
- No Playwright or browser-based smoke testing — `curl` is sufficient.
- No multi-environment (staging/prod) setup — single environment only.

## Design Considerations

- The Postgres connection string should be assembled from individual secret values (host, port, user, password, db name) rather than storing the full URL as a single opaque secret, to make credential rotation easier.
- Use `imagePullPolicy: Always` in the app Deployment so the `latest` tag always pulls the freshest image after each CI deploy.
- The SSH tunnel in GitHub Actions should forward both the k3s API server port (default `6443`) and the Traefik ingress port (`80`) so that both `helm`/`kubectl` and the smoke test can operate through the tunnel.

## Technical Considerations

- **Node.js:** 22 LTS (`node:22-alpine`)
- **Postgres:** 17 (Bitnami chart, image tag pinned to `17`)
- **pnpm:** The `Dockerfile` must install `corepack` and enable pnpm before installing dependencies.
- **Next.js standalone output:** `output: 'standalone'` in `next.config.ts` is required for the multi-stage Docker build to produce a minimal runtime image.
- **Prisma migrations:** The migration `Job` runs as a Helm `pre-install`/`pre-upgrade` hook with `hook-delete-policy: before-hook-creation`. Helm will wait for the Job to complete before proceeding with the Deployment rollout, so no explicit `kubectl` step is needed in CI.
- **kubeconfig for CI:** The kubeconfig stored in `secrets.KUBECONFIG` must have its `server` URL pointing to `localhost:<api-port>` since it will be used after the SSH tunnel is established on the runner.

## Success Metrics

1. `docker build` completes without errors and the final image is under 300 MB.
2. Trivy reports zero CRITICAL/HIGH CVEs in the final image layer.
3. `helm upgrade --install` completes without errors on the k3s cluster.
4. All pods reach `Running` state; readiness probe passes within 60 seconds of deployment.
5. HPA is present and reports correct `minReplicas: 1` / `maxReplicas: 3` configuration.
6. Smoke test exits 0 — HTTP 200 returned and body contains expected content.
7. Full GitHub Actions pipeline (build → scan → push → tunnel → migrate → deploy → smoke test) completes green on a push to `main`.
