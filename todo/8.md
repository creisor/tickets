# 8 - Connect GitHub Actions to k3s cluster via Tailscale

# Description

The GitHub Actions deploy workflow (`deploy.yml`) cannot reach the k3s cluster
because the runner has no access to the libvirt NAT network the cluster runs
on. The fix is to use Tailscale to join the runner to the same tailnet as the
k3s node during each workflow run.

## Approach

1. Install Tailscale on the k3s host node.
2. Add the `tailscale/github-action` step to the workflow — it joins the
   ephemeral runner to the tailnet using a reusable auth key.
3. Update the `KUBECONFIG` GitHub secret to point to the k3s node's Tailscale
   IP instead of `127.0.0.1`.

## Affected files

- `.github/workflows/deploy.yml` — add Tailscale connect step, remove SSH
  tunnel workaround
- `docs/self-hosted-runner.md` — document Tailscale setup on the k3s node and
  the new `TAILSCALE_AUTHKEY` secret
- `KUBECONFIG` GitHub secret — must be updated to use the Tailscale IP

## Notes

- The Tailscale auth key should be an **ephemeral, reusable** key so runner
  nodes auto-expire from the tailnet after the job completes.
- The k3s node only needs to be a Tailscale peer (no subnet routing required)
  since the runner only needs to reach port 6443 on that node.
- The `KUBECONFIG` secret must have `server: https://<tailscale-ip>:6443`.
